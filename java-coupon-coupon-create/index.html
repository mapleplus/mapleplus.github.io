  • 创建优惠券的主力是商家用户，按照淘宝、天猫非官方数据统计，商家数量已有近 2500 万。假设每个商家会创建 100 张优惠券（这个数据是折中记录，有些商家可能创建几十，有些商家可能会创建 1000 不止），那优惠券模板表就会接近 30 亿数据量。而 MySQL 数据库最大单表可承受 2000 万 数据量，因此需要分表；只分表就要分 120 张表，在电商这种需要高并发写入和海量数据查询处理的业务场景下，之前测过单一数据库压测瓶颈大概在 4000-5000/TPS ，这里选择 10000/TPS ，所以分两个库，每个库 60 张表。
# 如何选择分片键？
  • 与业务强关联
  • 选定之后不应该再做更改
  • 保证数据分散均匀
  • 优惠券和商家店铺产生强关联，用户也根据商家兑换优惠券，因此选择店铺编号 shopNumber 作为分片键
# 如何选择分片算法？

目前流行的两种算法：

  • HashMod
  • 时间范围
  • 早期商家较少，后期商家较多，选择时间范围，优惠券的数量会分布前多后少，所以选择 HashMod 算法

# 2. 优惠券创建责任链设计

# 责任链模式

  • 责任链模式（Chain of Responsibility Pattern）是一种行为设计模式，它允许将请求沿着一个处理链传递，直到链中的某个对象处理它。这样，发送者无需知道哪个对象将处理请求，所有的处理对象都可以尝试处理请求或将请求传递给链上的下一个对象

    代码通过 merchantAdminChainContext.handler 调用责任链模式对请求参数进行校验。责任链模式的引入使得校验逻辑模块化，每个校验规则可以独立为一个处理节点（如权限校验、参数合法性校验、库存校验等）。这种设计具有以下优势：
  • 解耦：校验逻辑与业务代码分离，新增或修改校验规则无需改动核心流程。
  • 灵活扩展：可通过动态增减责任链节点支持不同场景（如不同商户类型有不同的校验规则）。
  • 可维护性：每个校验节点的职责单一，便于单元测试和问题排查。

# 3. 缓存预热与延迟消息保障数据一致性

代码通过双保险机制确保数据库与缓存的最终一致性：

  • 缓存预热：将新创建的优惠券模板数据立即写入 Redis ，并设置缓存过期时间与模板有效期一致。这避免了首次查询时的缓存穿透问题，同时利用 Redis 自动过期机制清理无效数据。
    • 优惠券保存到 Redis 是使用 Hash 而不是 String ，因为 Hash 可以增加发行量。
    • 设置过期时间需要分布执行，为了避免极端情况，需要使用 Lua 脚本原子执行命令；并且可以设置内存淘汰策略辅助保证 Redis 内存空间的使用
  • RocketMQ 延迟消息：在优惠券有效期结束时，通过消息触发数据库状态更新与缓存清理操作。这里可以做扩展：在这里进行缓存数据的删除，就省略了缓存预热设置过期时间的业务复杂度。

# 4. 幂等处理

因为网络问题或者其他原因，用户可能多次发送创建同一个优惠券模板的请求到后端，避免无效、冗余的创建，需要做幂等处理。

  • 在 framework 模块设置 @DuplicateSubmit 注解，并定义 Spring AOP 环绕增强
  • 使用请求路径 + 用户 ID + 请求参数哈希作为分布式锁的 key，每次请求到来使用 tryLock 快速获取结果，处理幂等问题
    • 这里根据业务场景划分：
      • 若业务要求全局防重（如商品秒杀），需移除用户 ID，仅用接口路径和参数哈希。
      • 若仅限制用户对同一接口的提交频率（无论参数），可移除参数 MD5，改用固定 Key，（如 lockKey = user + path）。
  • try-finally 最后释放锁
  • Redisson 的看门狗机制可以很好的处理锁的续期
  • 极端情况下，如果宕机了锁无法释放怎么办？
    • Redis 宕机：这里做幂等校验，无法释放也没关系，可以设置内存淘汰策略优化内存使用
    • 业务服务宕机：不影响 Redis 的正常运行。

# 4. 其他值得注意的细节

  • 布隆过滤器防缓存穿透：新增模板时将其 ID 加入布隆过滤器，后续查询时通过过滤器快速判断 ID 是否存在，避免无效查询直接穿透到数据库。
  • 日志记录与上下文处理：使用美团的 mzt-biz-log ，手动设置 LogRecordContext 中的 bizNo，确保日志能正确关联业务 ID。
  • 为什么使用现有框架而不是重复造轮子？效率提升、质量保障、持续更新

# 5. 潜在优化点

  • 缓存序列化：将对象转为 Map 时，若字段包含复杂类型（如日期），可能需显式处理格式，避免反序列化异常。
  • 消息可靠性：需确保 RocketMQ 消息的可靠投递（如事务消息、消费端幂等处理），防止消息丢失导致状态不一致。
  • 布隆过滤器维护：长期运行后需考虑布隆过滤器的误判率上升问题，可结合定时重建或计数布隆过滤器优化。
