# 百万级优惠券分发技术实践（单服务节点）

# 需求分析

在大型电商平台中，优惠券可以给与用户许多优惠，促进商品的营销，助力企业利润的增长，使用非常频繁。因此，在淘宝、美团、饿了么等平台都会有商家 / 平台主动推送商品优惠券的功能。而优惠券的分发会涉及海量用户，分发过程需要保证服务的高可用、高性能，因此进行数据库表的设计、中间件的选型十分重要。

# 优惠券分发业务是否要做限领？

需要做限领，这是保障业务健康运转的核心机制，具体原因如下：

# 1. 限领的核心价值
场景 问题 限领作用
防止薅羊毛 黑产批量领取优惠券，导致正常用户无法参与活动，商家成本失控。 限制单个用户领取次数（如每人限领 1 次），降低恶意行为影响。
精细化运营 高价值优惠券被少数用户大量囤积，无法触达目标用户。 通过限领规则（如按用户等级、行为标签限领）精准投放，提升转化率。
成本控制 无限制领取可能导致预算超支，尤其对满减、折扣类优惠券。 结合库存总量和限领规则，实现成本可控。
用户体验 用户重复领取相同优惠券（如无感分发），造成信息干扰。 限领规则避免重复发放，提升用户感知价值。

# 2. 是否与用户主动领取的优惠券做数据隔离？

建议做数据隔离，但需根据业务场景灵活设计：

# 数据隔离方案
设计方向 实现方式 优势 适用场景
物理隔离 分发的优惠券和用户主动领取的优惠券存储在不同表（如 auto_coupon 和 manual_coupon ）。 规则独立管理，逻辑清晰；限领检查无需跨数据源，性能高。 高并发系统、分发与领取逻辑差异大的场景（如无感发券 vs 用户抢券）。
逻辑隔离（目前实现方式） 同一表中增加 source 字段（如 source=PLATFORM 代表分发， source=USER 代表用户主动领取）。 数据统一存储，便于全局统计；通过字段过滤实现限领规则隔离。 中小型系统，需简化数据模型。
混合隔离 核心字段（如优惠券模板）统一管理，用户领取记录按来源隔离。 平衡灵活性与性能，支持按来源动态配置限领规则（如分发的券不占用用户主动领取的限额）。 需要动态规则配置的场景。
# 限领规则设计

无论是否隔离数据，均需结合业务需求设计限领维度：

限领维度 规则示例 实现方式
全局限领 用户在整个平台最多领取 3 张该券（无论来源）。 查询用户所有来源的该券领取记录，判断总数是否超限。
按来源限领（目前实现方式） 用户通过平台分发渠道最多领取 1 张，主动领取店铺渠道最多 2 张。 根据 source 字段分别统计，隔离检查。
按活动限领 用户在同一活动中（如双 11）只能领取 1 张该券。 关联活动 ID，限制同一活动下的领取次数。
动态限领 新用户可领 2 张，老用户只能领 1 张。 结合用户标签或行为数据，动态计算限领值。

# 3. 技术实现建议

# 限领检查流程（以逻辑隔离为例）
# 无感分发与主动领取的限领策略对比
场景 限领策略 技术实现
无感分发 通常按用户标签批量发放，不占用用户主动领取的限额（如分发票券限额 = 5，用户主动领取限额 = 2）。 数据隔离 + 独立限领规则（通过 source 区分）。
用户主动领取 用户手动点击领取，需实时检查全局或活动限领规则。 高并发下用 Redis 计数（如 INCR 命令），保证原子性。
混合场景 用户既有无感分发票券，又有主动领取的票券，需区分统计。 使用联合唯一索引（如 user_id + coupon_template_id + source ）防重。

# 4. 不隔离数据的风险

若不隔离数据，需额外处理以下问题：

  1. 限领规则冲突
    • 用户因无感分发已占满限额，导致无法主动领取，引发投诉。
  2. 运营灵活性下降
    • 无法针对不同来源设置差异化规则（如分发票券不限领，主动领取限领 1 次）。
  3. 数据统计失真
    • 难以区分用户行为来源，影响后续运营分析。

# 示例：隔离数据后的限领规则配置

结论：限领是优惠券系统的核心设计，数据隔离能显著提升规则灵活性和系统可维护性。通过 逻辑隔离（ source 字段） + 动态限领规则 实现业务需求，技术层面需结合 Redis 计数和数据库唯一索引保障高并发场景下的准确性。

# 技术选型

# MySQL

用于分发任务、用户优惠券、失败记录数据持久化。

# Redis

需要使用 Redis 吗？每引入一个中间件，系统的复杂度都会提升，需要解决数据一致性、服务稳定等问题。经过分析，优惠券分发的这些场景是需要的：

  1. 断点续传。任务由平台 / 商家发起后，需要进行分发任务进度存储，防止服务宕机后重新进行分发任务导致的重复分发。举个例子，在双十一、国补等一些活动即将开始的时候，有的商品比如电脑、手机、冰箱等要做优惠券的限领。

  1. 保存分发用户记录。分发的时候，涉及进度存储，那么也会涉及到已经分发的用户信息存储，在后续服务保存用户优惠券记录到数据库的时候，可以从缓存快速获取用户信息进行记录保存。
  2. 保存用户优惠券。在其他服务如订单、购物车结算等服务时，海量用户会对自己的优惠券进行查询，如果不使用缓存，在 MySQL 直接查询耗时比较长（优惠券也会保存在 Redis 中），为了应对秒杀等海量并发的快速响应，因此需要 Redis 对用户优惠券记录保存，直接在缓存获取数据进行响应的业务处理。

# RocketMQ

  1. 优惠券分发一定是需要消息队列的，不论是立即分发还是定时分发，在商家 / 平台发起任务后，由后台执行任务，执行完成之后通过邮箱 / 手机短信 / 站内信通知商家 / 平台人员即可，并且还可以通知用户及时使用券。
  2. 分发涉及到 Redis、MySQL 相关的操作，海量数据，如果下游消费者久久不能响应，RocketMQ 会发起重试消费，导致分发业务重发，因此需要对下游的消费者再解耦，使用双消费者模式保障分发服务稳定。

# 线程池配置

因为服务要进行失败报告记录，为了精确知道用户信息 Excel 分发的成功与否，或者需要对账系统统计成功和失败的比例方便平台查看，因此需要统计 Excel 的用户记录行数，用户信息 Excel 由数仓得出。而如果我们提交任务后因为需要解析统计巨量 Excel 行数导致响应变得极慢，是不合理的，因此使用线程池异步解析 Excel 行数，并持久化到分发任务表中。

# corePoolSize：CPU 核心数
# maximumPoolSize：CPU 核心数 * 2
# keepAliveTime：60s
# 时间单位：秒
# 任务队列：指定大小的 ArrayBlockingQueue
# 拒绝策略：自定义拒绝策略，日志打印，平台告警，丢弃任务

那么，我们直接拒绝任务后是不是任务就不执行了？当然不是，还做了兜底机制，在线程异步后，使用 Redisson 发送延时队列，20 秒后进行再检测刷新行数。

# 流程概述

  1. 参数校验：验证请求参数合法性。
  2. 优惠券模板校验：通过 couponTemplateService （已解决缓存击穿、穿透预防方案）检查模板是否存在。
  3. 构建任务实体：转换请求参数为 CouponTaskDO ，设置批次 ID、操作者、店铺号、发送类型（立即 / 定时）。
  4. 数据库插入：持久化任务记录。
  5. 异步刷新 Excel 行数：提交线程池任务，读取 Excel 行数并更新任务记录。
  6. 延迟队列兜底：通过 Redisson 延迟队列（20 秒后）再次刷新行数，防止应用宕机。
  7. 立即发送处理：若任务类型为立即发送，更新任务状态为 “进行中”，发送 RocketMQ 消息触发分发。
  8. 定时分发使用定时任务执行

# 分发任务消费（消费者一）

# 核心设计目标（Redis 层面操作）
  1. 高效处理海量数据：通过分批处理、Lua 脚本原子操作、Redis 进度缓存和本地进度缓存提升性能。
  2. 断点续传能力：通过 Redis 记录分发进度，任务中断后可从断点恢复。
  3. 库存与防超发控制：利用 Redis Lua 脚本实现库存扣减与用户记录的原子操作。
  4. 异常处理与监控：失败记录落库，进度实时保存，消息队列解耦分发。
# 核心流程分解
# 1. 初始化阶段
  • 断点加载：
    • 从 Redis 读取任务进度键，恢复已处理的行数 lastSavedRowCount 。
    • 设置 当前行数 = 已处理的行数 + 1 ，跳过已处理行，实现断点续传。
# 2. 逐行处理（invoke 方法）
  • 行过滤：
    • 若当前行 rowCount 小于等于 lastSavedRowCount ，直接跳过（已处理）。
  • Lua 脚本执行：
    • 脚本加载：通过单例模式缓存 Lua 脚本，减少 IO 开销。
    • 参数传递：
      • couponTemplateIdKey ：优惠券键（如 coupon:template:{id} ）。
      • userSetKey ：用户领取记录键（如 task:users:{taskId} ）。
      • userRowNumMap ：用户 ID 与行号（用于记录进度和用户关联）。
    • 原子操作：脚本内完成库存扣减与用户记录写入，确保一致性。
  • 库存不足处理：
    • 若 Lua 返回库存不足，记录失败原因到 couponTaskFailMapper ，保存当前进度。
  • 批量分发触发：
    • 检查 batchSize 是否达到 BATCH_USER_COUPON_SIZE （如 100000）。
    • 满足条件时，构造分发事件 CouponTemplateDistributionEvent ，发送至消息队列。
    • 事件中标记 distributionEndFlag 区分中间批次与最终批次。
  • 进度保存：
    • 批量保存：每处理 BATCH_SAVE_PROGRESS_SIZE （如 100000）行更新一次 Redis 进度和本地进度缓存。
    • 立即保存：库存不足等异常或结束时强制保存。
# 3. 结束处理（doAfterAllAnalysed）
  • 强制保存进度：确保最后一行进度保存。
  • 发送结束事件：构造分发结束事件，标记 distributionEndFlag = TRUE ，触发后续收尾逻辑。

# 执行分发消费（消费者二）

该消费者类处理优惠券分发事件，MySQL 层面操作，核心逻辑分步骤梳理如下：

# 消息处理入口： onMessage 方法
  • 步骤 1.1：消息解析
    • 接收 RocketMQ 消息，提取 CouponTemplateDistributionEvent 事件对象。
  • 步骤 1.2：判断是否为中间批次
    • 若事件未结束（ distributionEndFlag=false ），调用 decrementCouponTemplateStockAndSaveUserCouponList 处理当前批次用户。
    • 异常处理：捕获异常并记录日志，避免消息消费失败。
# 核心方法： decrementCouponTemplateStockAndSaveUserCouponList

  • 步骤 1.2.1：扣减库存（指定数量进行批量扣减）

    • 调用 decrementCouponTemplateStock ，尝试扣减数据库库存。
    • 条件扣减库存：通过 couponTemplateMapper.decrementCouponTemplateStock 更新库存。
    • 递归重试：
      • 可能优惠券库存不满足用户数量刚好扣减为 0 的数量，如果到最后批次需要扣减 4986，但是优惠券库存由商家自定义设置，最后只剩 2000，所以若库存不足，查询当前实际库存，递归扣减剩余量。
      • 如果我们使用多个服务节点集群，需要考虑并发控制，那么会不会因为多次库存不足导致深度递归，栈的深度过大？答案是不会的：
        • 如果库存不足，商家只发起一个分发任务：第一层，扣减 5000，不足，查询剩余库存 3000；第二层，扣减 3000，完成。
        • 如果商家发起第二个，第三个任务...... 那么后面的任务可能有部分批次扣减成功，但是剩余的批次肯定都是因为库存早已归 0 扣减失败，每个服务节点批量扣减的时候都是要指定扣减数量的，最多递归 2 层就会结束扣减动作，保存失败记录。
    • 库存因归 0 不足：直接返回，不进行后续操作。

  • 步骤 1.2.2：从 Redis 获取用户列表

    • 弹出指定批次数量（ decremented ）的用户 ID 和行号（ userIdAndRowNumList ）。
    • 空值处理：若 Redis 无数据，可能是 Redis 宕机丢失数据， 恢复当前批次扣减的库存，记录日志并返回，等待后续重新提交任务进行分发。
    • 遍历用户列表，构建 UserCouponDO 对象，设置有效期、领取时间等字段，构建用户优惠券集合 userCouponList 。

  • 步骤 1.2.3：批量插入用户券记录

    • 调用 batchSaveUserCouponList ，批量插入数据库。
    • 唯一索引冲突：捕获 BatchExecutorException ，逐条重试插入，记录失败用户到 coupon_task_fail 表，并移除掉用户集合 userCouponList 中失败的用户实体，** 恢复当前批次扣减的库存，** 后续仅保存分发成功的用户记录
      • 为什么会发生索引冲突？这里是为了避免商家提交多次相同的任务，扣减库存还没归 0 但是已经到达了保存用户优惠券的时刻，比如，A 任务扣减 5000，保存了本批次的用户，库存足够，B 任务扣减 5000，又保存用户，这时就会发生冲突。因此需要处理批量插入的冲突问题。
    • 事务管理：事务隔离级别至少是 RC。 executeTransaction.hasUserGetCoupon (each)
    • 唯一索引检查时机：在事务提交时统一检查约束（如唯一索引），而非每次插入时立即检查。
    • 事务内的中间状态：在事务提交前，数据库允许事务内部暂时违反唯一索引约束，但最终提交时会回滚整个事务（如果存在冲突）。
    • 如果第一次批量保存数据库失败后，批量保存的一些记录会在事务缓冲区中，如果在同一个事务下，会查到数据库中本不存在的记录。我们保存用户已领取优惠券的失败记录，需要是数据库中真实存在的，所以不使用事务的方式查询

  • 步骤 1.2.4：缓存用户 - 优惠券关系

    • Lua 脚本执行：从用户集合 userCouponList 将用户 ID 与当前优惠券 ID 关系存入 Redis，设置过期时间。
    • 原子性保障：通过 Lua 脚本确保缓存操作的原子性。

  • 步骤 1.3：处理最终批次

    • 条件： distributionEndFlag=true ，表示 Excel 用户数据已处理完毕。
    • 操作：
      1. 检查 Redis 剩余用户：通过 userSetKey 获取剩余待发放用户数量。
      2. 扣减库存并保存用户券：调用 decrementCouponTemplateStockAndSaveUserCouponList 。
      3. 处理未发放用户：若 Redis 仍有用户数据（如因重复领取 <==> 库存不足），记录失败原因到 coupon_task_fail 表。
      4. 生成失败记录 Excel：
        • 分页查询失败记录，避免深分页问题，使用 maxId 作为游标。
        • 使用 EasyExcel 上传 OSS 平台路径 excelPath ）。
        • 记录失败行数，Excel 生成结束后补齐缓存的优惠券库存，保证和数据库的数据一致性
      5. 更新任务状态：标记任务为成功，设置完成时间。
      6. 发送分发状态消息到下游：执行 邮箱 / 短信 / 站内信 通知商家 或 平台人员 或 提醒用户及时使用优惠券
